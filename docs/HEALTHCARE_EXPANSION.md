# 헬스케어 애플리케이션 전환 기획 (초안)

> 상태: **초안**. 확정된 결정이 아니라 Figma 디자인과 구현을 시작하기 위한 논의 기반입니다.
> 작성 시점의 코드 실측을 근거로 하며, 미정 항목은 `<!-- TODO -->`로 표시합니다.

## 1. 현재 위치

`k-calAI-RN`은 지금 **식단 분석 기능 하나짜리 앱**입니다. 헬스케어 앱의 일부처럼 보이도록 문구를 맞췄을 뿐, 실제로는 그렇지 않습니다.

### 실제로 존재하는 것 (코드 실측)

| 계층 | 있는 것 |
|------|---------|
| 화면 | `auth`(휴대폰 인증), `(tabs)/index`(분석). `(tabs)/explore`(개발자용 진단)는 2026-07-13 삭제 |
| 서버 API | `/api/auth/{signup,login}/{request-code,verify}`, `/api/predict`, `/api/nutrition/estimate` (레거시 `/api/gpt-predict`·`/api/s3/*`는 2026-07-12 제거) |
| DB 테이블 | `users`, `phone_verification_codes`, `auth_sessions` |
| 세션 | 메모리에만 존재. 새로고침하면 로그아웃 |

### 없는 것

**식단 기록이 저장되지 않습니다.** 분석 결과는 화면 상태로만 존재하고 앱을 나가면 사라집니다. 사용자·음식·섭취량을 잇는 테이블이 하나도 없습니다.

즉 "헬스케어 앱으로 둔갑"의 첫 관문은 UI가 아니라 **데이터 모델**입니다. 이걸 정하지 않고 Figma를 그리면 화면이 저장할 수 없는 것을 보여주게 됩니다.

## 2. 제품 정의

### 한 줄

> 사진 한 장으로 식사를 기록하고, 하루 섭취량과 목표를 눈으로 확인하는 앱.

### 핵심 루프

```
목표 설정(1회)  →  끼니마다 사진 기록  →  하루 진행률 확인  →  주간 추이 확인
```

이 루프가 돌지 않으면 나머지 기능은 의미가 없습니다. **첫 릴리즈는 이 루프만 완성합니다.**

### 비목표 (기존 문서 유지)

- 의료 진단·질병별 식단 처방을 제공하지 않는다.
- 사진 한 장으로 정확한 중량·영양소를 확정하지 않는다.
- AI 모델 성능 비교를 사용자 경험의 전면에 노출하지 않는다.

## 3. 화면 목록 (Figma 작업 단위)

기존 화면은 **유지·개편**, 신규는 **새로 그림**입니다.

### 온보딩

| # | 화면 | 상태 | 내용 |
|---|------|------|------|
| 1 | 스플래시 / 로그인 진입 | 개편 | 현재 `auth.tsx`가 곧바로 인증 폼입니다. 앱 소개 1스텝이 앞에 필요합니다 |
| 2 | 휴대폰 인증 | 유지 | 현재 화면 재사용. `dev_code` 표시는 프로덕션에서 제거 |
| 3 | 프로필 입력 | **신규** | 성별, 생년(또는 나이), 키, 몸무게, 활동량 |
| 4 | 목표 설정 | **신규** | 체중 감량/유지/증량 → 일일 목표 칼로리 자동 산출 + 수동 조정 |

### 메인 (탭)

| # | 탭 | 상태 | 내용 |
|---|-----|------|------|
| 5 | **홈 (오늘)** | **신규** | 오늘 섭취/목표 진행률 링, 남은 칼로리, 끼니별(아침·점심·저녁·간식) 카드, 빠른 기록 버튼 |
| 6 | **기록 (분석)** | 개편 | 현재 `(tabs)/index.tsx`. 사진 → 후보 선택 → 칼로리 확인까지는 유지하고, **끝에 "기록 저장" 단계 추가** |
| 7 | **추이** | **신규** | 주/월 섭취 칼로리 그래프, 체중 변화, 목표 달성률 |
| 8 | **내 정보** | **신규** | 프로필·목표 수정, 로그아웃, 개인정보 정책, 앱 정보 |

`(tabs)/explore.tsx`("상태" 탭, API 엔드포인트를 노출)는 **완전히 삭제했습니다** (2026-07-13). 서버 URL은 사용자에게 보여줄 정보가 아닙니다 — 로그인 화면·기록 화면의 '연결 서버' 카드도 함께 제거했습니다.

### 기록 플로우 상세 (6번 개편의 핵심)

```
사진 촬영/선택
  → 분석 (POST /api/predict)
  → 후보 3개 표시
  → 후보 선택
  → 칼로리·영양 추정 (POST /api/nutrition/estimate, 식약처 DB)
  → ★ 기록 확정 화면 (신규)
       · 음식명 직접 수정
       · 1인분 기준 / 양 조절 (0.5인분, 1인분, 1.5인분 …)
       · 끼니 선택 (아침/점심/저녁/간식)
       · 여러 음식 추가
       · 최종 칼로리 확인
  → 저장 → 홈으로 복귀, 진행률 갱신
```

★ 표시된 "기록 확정 화면"이 **이번 전환의 최대 신규 화면**입니다. 여기서 저장이 일어납니다.

### 공통 요소

- 하단 탭 4개 (홈 / 기록 / 추이 / 내 정보)
- 빈 상태(기록 없음), 로딩, 오류 배너 — 이미 오류 배너 패턴은 존재
- 추정치 고지 문구 ("AI 추정값이며 실제와 다를 수 있습니다")

## 4. 데이터 모델 (신규 필요)

현재 서버에는 `users`, `phone_verification_codes`, `auth_sessions`뿐입니다. 아래가 최소 추가분입니다.

| 테이블 | 목적 | 주요 컬럼(안) |
|--------|------|---------------|
| `user_profiles` | 신체 정보 | `user_id`(FK), `sex`, `birth_year`, `height_cm`, `weight_kg`, `activity_level` |
| `user_goals` | 목표 | `user_id`, `goal_type`(loss/maintain/gain), `target_kcal`, `target_weight_kg`, `started_at` |
| `meal_logs` | 끼니 | `user_id`, `logged_at`, `meal_type`(breakfast/lunch/dinner/snack), `photo_url`(nullable), `total_kcal` |
| `meal_items` | 끼니 안의 음식 | `meal_log_id`(FK), `food_label`, `serving_ratio`, `kcal`, `source`(ai/manual), `confidence` |
| `weight_logs` | 체중 추이 | `user_id`, `measured_at`, `weight_kg` |

### 설계 원칙

- 시간은 **timezone-aware UTC** 저장. 화면 표시만 로컬 변환 (기존 `docs/DESIGN.md` 규칙 유지).
- `meal_items.source`로 AI 추정과 사용자 수정을 구분합니다. **모델 개선의 근거가 됩니다.**
- `meal_logs.total_kcal`은 `meal_items` 합계의 캐시입니다. 조회 성능용이며 단일 진실은 `meal_items`.
- 사진 원본 저장 여부는 **개인정보 결정**입니다. 저장한다면 `photo_url`이 S3 키를 가리킵니다. 서버에 `/api/s3/*`가 이미 있습니다.

## 5. 서버 작업 (`kcalAI-model`)

### 선행 조건 — 이게 없으면 아무것도 저장 못 합니다

**세션 토큰 검증이 구현되어 있지 않습니다.** `AuthSession.token`을 발급만 하고 검증하는 코드가 없으며, 앱도 `Authorization` 헤더를 붙이지 않습니다. 개인 데이터를 다루려면 이걸 먼저 해야 합니다.

| 순서 | 작업 | 비고 |
|:----:|------|------|
| 1 | `Depends(get_current_user)` 인증 의존성 | `auth_sessions` 조회 + `expires_at`/`revoked_at` 검사 |
| 2 | 앱이 `Authorization: Bearer <token>` 첨부 | 두 저장소 동시 변경 |
| 3 | 세션 영속화 (`expo-secure-store`) | 지금은 새로고침하면 로그아웃 |
| 4 | `POST /api/auth/logout` | `revoked_at` 갱신 |
| 5 | 프로필·목표·식단 기록 CRUD | 위 데이터 모델 기준 |
| 6 | Alembic 도입 | `create_all`은 컬럼 변경을 반영하지 않습니다 |

### 함께 정리할 것 (기존 `CLAUDE.md`의 알려진 문제)

- `AUTH_INCLUDE_DEV_CODE` 기본값 `true` → 운영에서 인증번호 노출
- `AUTH_CODE_PEPPER` 기본값 그대로
- `/api/s3/*`의 `detail`에 boto3 내부 예외 노출 (14곳)
- `runs/` 70MB가 저장소에 커밋됨

## 9. 온보딩

가입 직후 한 번만 통과하는 사슬입니다. **앞 단계가 뒤 단계의 입력을 결정**하므로 순서를 바꿀 수 없습니다.

**0단계 — 민감정보 동의**는 인증 직후, 온보딩 진입 전에 받습니다. 동의 이력은 `user_consents`에 **버전과 시각**을 남깁니다. 약관이 바뀌었을 때 누가 무엇에 동의했는지 증명해야 하기 때문입니다.

| 단계 | 화면 | 필수 | 수집 항목 |
|:----:|------|:----:|-----------|
| 0 | 민감정보 수집 동의 | 필수 | `user_consents.kind = 'sensitive_health'` |
| 1 | 키·몸무게·생년 | 필수 | BMR 계산의 입력. 이게 없으면 목표를 산출할 수 없습니다 |
| 2 | 혈액형 | 선택 | `blood_type`, `rh` |
| 3 | 질병 유무 | 선택·복수 | 당뇨 / 임신 / 신장 질환 / 암 치료 중 / 고혈압 / 해당 없음 |
| 4 | 알러지 | 선택·복수 | 땅콩·우유·갑각류·계란·밀·대두·복숭아 등 |
| 5 | 목표 확인 | 필수 | 자동 산출된 `target_kcal`을 보여주고 수정을 허용 |

2~4단계는 건너뛸 수 있습니다. 건너뛰면 식단 추천이 일반 가이드로 격하된다는 것을 그 자리에서 알립니다. 동의하지 않으면 2~4단계를 아예 보여주지 않습니다.

## 10. 그룹

한 사용자가 **여러 그룹에 속할 수 있습니다** (N:N). `groups.kind`로 성격을 나눕니다.

| kind | 예 | 비고 |
|------|-----|------|
| `family` | 우리 가족 | |
| `couple` | 연인 | |
| `friends` | 친구 | |
| `challenge` | 1월 다이어트 모임 | 그룹 목표 달성률을 보여줍니다 |

그룹은 **내 정보 하위가 아니라 홈에서 진입**합니다. 매일 보는 자리에 있어야 모임이 굴러갑니다.

### 반려동물은 그룹 종류가 아닙니다

**보호자 1명 : 반려동물 N마리** 구조입니다. 반려동물은 `pets` 테이블의 독립 엔티티이며 `owner_id`로 보호자를 가리킵니다. 로그인하지 않고, 메인 사용자 층도 아닙니다.

- 그룹에 참여시킬 때는 `group_pets`로 붙입니다. 사람 멤버(`group_members`)와 테이블을 분리합니다 — 다형성 FK를 피하기 위해서입니다.
- 급여 기록은 `meal_logs`가 아니라 `pet_feeding_logs`에 남깁니다. `meal_logs.user_id`는 사람 FK입니다.
- **Mifflin-St Jeor가 적용되지 않습니다.** 개는 RER = `70 × 체중^0.75`, MER = RER × 활동계수(중성화·연령·비만도에 따라 1.2~1.8)입니다. MVP에서는 **급여량(g)만 기록**하고 칼로리 산출은 다음 단계로 미룹니다.

| iOS | Apple 건강 (HealthKit) | 네이티브 모듈. **Expo Go에서 동작하지 않습니다** |
| Android | Health Connect | 삼성 헬스가 여기에 데이터를 씁니다 |
| 웹 | **없음** | 브라우저에 대응 API가 없습니다. 연동 카드를 비활성으로 두고 이유를 표시합니다 |

가져오는 것: 걸음 수, 활동 소모 kcal, 체중. 활동량으로 **TDEE를 보정**합니다.

**삼성 헬스는 자체 SDK 대신 Health Connect를 경유합니다.** 파트너 승인이 필요 없고 Android 표준입니다.

## 12. 식단 추천

**처방이 아니라 회피 필터입니다.** (아래 13장 참조)

- 입력: 남은 kcal, `user_conditions`, `user_allergies`, 끼니 종류
- 동작: 알러지·질병에 해당하는 재료를 **제외**하고 후보를 고른 뒤, 남은 칼로리 안에 맞춥니다
- 모든 추천에 **"AI 추정값이며 의학적 조언이 아닙니다"**를 고지합니다
- 사진 분석 결과에 제외 재료가 보이면 기록할 때 경고합니다
- 추천 결과는 `diet_recommendations`에 캐시해 LLM 재호출을 막습니다

## 13. v2 결정 사항 (2026-07-09 확정)

| # | 결정 | 내용 |
|---|------|------|
| 1 | **질병별 식단 추천** | **MVP/PoC로 구현한다.** 2장의 비목표는 MVP 동안 유보한다. **의학 전문가가 합류해 감수**하고, 법률 검토는 그 이후 단계다. 단 모든 추천 화면의 **"AI 추정값이며 의학적 조언이 아닙니다" 고지는 전문가 감수 전까지 제거하지 않는다** |
| 2 | **민감정보** | **가입 시 동의 플로우(0단계)를 넣는다.** 동의 이력은 `user_consents`에 버전·시각과 함께 남긴다. 저장은 `user_health_profiles`로 분리한다 |
| 3 | **헬스 앱 연동** | **기획만 하고 구현은 미룬다.** development build 전환이 필요하므로 MVP 이후다 |

### 프로덕션 전 남는 것 (MVP에서는 막지 않는다)

- `/api/predict`와 `/api/s3/*`가 **무인증 공개**다. 닫아야 한다.
- `user_health_profiles`의 컬럼 암호화 여부를 정해야 한다.
